# AGENTS.md

## 项目定位

本项目是一个**仅供个人本机使用**的测试智能体工作台，目标是协助测试人员完成以下工作：

- 解析需求文档、截图和页面信息，输出结构化测试分析
- 结合 OpenAPI 文档生成 UI/API 自动化开发任务包
- 根据 bug 描述推荐回归场景
- 执行本地自动化测试，生成报告与云效缺陷草稿

## 开发原则

1. 代码优先保证复用性、抽象性与可读性，避免为单点需求堆积临时代码。
2. 较大、较复杂、高风险或边界不清的需求，先分析方案与约束，再开始实现。
3. 关键业务节点必须补齐中文注释，优先说明业务意图、前置约束和保留原因。
4. 本项目为本地单机工具，优先考虑稳定性、可调试性和可回退能力，不追求过度自动化。

## 架构边界

- `agents/`
  - 只负责结构化智能推理。
  - Agent 不直接读写数据库、不直接调用 UI 组件。
- `controller/`
  - 负责桌面端工作流编排。
  - 页面层只能调用 controller 暴露的接口。
- `services/`
  - 放纯工程能力：OpenAPI 解析、测试执行、报告生成、场景检索、产物收集。
  - 非智能任务不要放入 Agent。
- `adapters/`
  - 放模型、云效、密钥等外部系统适配。
  - 所有第三方集成都必须经过 adapter，避免散落在业务代码中。
- `store/`
  - 负责 SQLite 和统一配置读写。
- `ui/`
  - 只负责展示和触发，不承载业务推理。

## 编码约束

- Python 版本：`>=3.11`
- 默认使用 ASCII；仅在中文文档或中文注释场景使用非 ASCII。
- 新增类型或 Agent 输出，必须使用 Pydantic 模型定义。
- 页面线程不能直接执行长耗时任务，必须通过后台 worker 触发。
- 业务异常需要转成可操作的错误信息，不能静默吞掉。
- 配置和密钥不能散落到多个文件；统一通过 `ConfigStore` 和 `SecretStore` 管理。

## UI 组件规范

- 页面层新增下拉选择器时，统一使用 `src/local_test_agent/ui/widgets.py` 中的 `AppSelect`，不要直接实例化 `QComboBox`。
- 下拉控件的尺寸配置统一走 `configure_combo_input(...)`，不要在页面内单独覆盖样式，避免再次出现系统原生 popup 与项目主题割裂。
- 若需要新增新的选择类控件形态，应先沉淀到 `ui/widgets.py` 做成可复用组件，再在页面里接入，不要在单页里写一次性实现。

## 配置约束

- 非敏感配置统一保存到 `data/app_config.json`
- 敏感信息优先保存到系统钥匙串；若不可用，再退化到本地密钥文件
- 不允许把真实 API Key、Token、Cookie、账号密码写入仓库

## 测试约束

- 新增功能至少补一个对应的 `pytest` 用例
- 若改动配置、持久化、执行链路，必须补回归测试
- 保持 `./.venv/bin/pytest` 可通过
- 若新增 UI 页面，至少保证 controller 层可测，必要时补离屏实例化验证

## 提交前检查

```bash
./.venv/bin/pytest
python3 -m py_compile src/local_test_agent/**/*.py
```

## 禁止事项

- 不要把 UI 逻辑写进 service 或 agent
- 不要让 Agent 直接修改业务仓库测试代码
- 不要在缺陷未确认时直接远程自动提单
- 不要在未说明业务原因时引入重量级基础设施
